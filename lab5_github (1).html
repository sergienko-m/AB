Лабораторна робота 5

Розширений аналіз вимог. Моделювання.
Програмна система: GitHub

1. Вступ

У лабораторній роботі виконано розширений аналіз вимог до платформи GitHub та побудовано шість UML-діаграм відповідно до стандарту UML 2.5. Діаграми описують систему з різних точок зору:

  • Діаграма варіантів використання — взаємодія акторів із системою.
  • Діаграма класів — статична структура сутностей та їх зв'язки.
  • Діаграма послідовності — динаміка взаємодії компонентів при ключовому сценарії.
  • Діаграма діяльності — алгоритм CI/CD pipeline у GitHub Actions.
  • Діаграма станів — життєвий цикл Pull Request.
  • Діаграма компонентів — архітектура платформи на рівні підсистем.

2. Діаграма варіантів використання

Діаграма показує чотирьох акторів та десять варіантів використання. Розробник клонує репозиторій, створює гілки та коміти, відкриває Pull Request. Тімлід проводить code review та зливає PR. DevOps налаштовує та запускає GitHub Actions. Власник організації керує доступом та переглядає аналітику. Відношення «include» між «Відкрити PR» та «Провести Code Review» означає, що code review є обов'язковою частиною процесу.

Рис. 1 — Діаграма варіантів використання GitHub

3. Діаграма класів

Діаграма відображає вісім ключових класів системи. Repository — центральна сутність: агрегується в Organization (1 до 0..*), містить Branch (1 до 1..*) через композицію, має асоціації до PullRequest, Issue та Workflow. Branch містить Commit (0..*). Перелічення PRStatus описує стани: DRAFT, OPEN, MERGED, CLOSED. Кожен клас має атрибути з позначками видимості (+/−/#) та операції.

Рис. 2 — Діаграма класів GitHub

4. Діаграма послідовності

Діаграма ілюструє сценарій створення та злиття Pull Request у 12 кроків між п'ятьма об'єктами. Після git push GitHub UI створює PR у PR Service, який автоматично тригерує CI. Actions Runner виконує тести і повертає статус. PR Service сповіщає Тімліда, який виставляє approve. Після злиття тригерується deploy workflow.

Рис. 3 — Діаграма послідовності: створення та злиття Pull Request

5. Діаграма діяльності

Діаграма описує CI/CD pipeline від git push до deploy. Після тригерування workflow три гілки виконуються паралельно: lint, unit-тести та сканування вразливостей. Перший вузол рішення: пройшли всі перевірки? При невдачі — PR позначається FAILED. Другий вузол: отримано Code Review approve? При відсутності — очікування. Після обох умов виконується злиття та deploy.

Рис. 4 — Діаграма діяльності: CI/CD pipeline GitHub Actions

6. Діаграма станів

Діаграма моделює 6 станів Pull Request. DRAFT — чернетка. OPEN — публічна, тригерує CI. IN REVIEW — надіслана рев'юерам. CHANGES REQUESTED — потрібні виправлення; після нового push повертається в OPEN. APPROVED — кнопка merge активна. Фінальні стани: MERGED та CLOSED.

Рис. 5 — Діаграма станів: життєвий цикл Pull Request

7. Діаграма компонентів

Чотирирівнева архітектура GitHub. Клієнтський рівень: Web UI (React), GitHub CLI (Go), Mobile App, IDE Plugin. API Gateway: REST v3 та GraphQL v4, rate limiting 5000 req/год. Сервісний рівень: Git Service, PR Service, Actions Engine, Security Service (CodeQL/Dependabot), Auth Service (OAuth2/SAML). Зберігання: Git Object Store (geo-replicated), Metadata DB (MySQL), Cache (Redis), Event Queue (Kafka).

Рис. 6 — Діаграма компонентів: архітектура GitHub

8. Висновок

Побудовано шість UML-діаграм, які описують GitHub з різних точок зору:

  • Структурний аналіз (діаграми класів та компонентів): 8 класів, 4-рівнева мікросервісна архітектура з 5 сервісами.
  • Поведінковий аналіз (варіанти використання): 4 актори та 10 сценаріїв з відношеннями «include».
  • Динамічний аналіз (послідовності та діяльності): 12-кроковий PR workflow, CI/CD pipeline з 3 паралельними гілками.
  • Станова поведінка (діаграма станів): 6 станів PR з 8 переходами.

UML-моделювання підтвердило, що ключова архітектурна характеристика GitHub — поєднання мікросервісної архітектури з подієво-орієнтованою взаємодією через Event Queue.
